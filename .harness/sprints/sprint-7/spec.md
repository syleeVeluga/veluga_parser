# Feature Spec: Enhanced PDF-to-Markdown Conversion with Page Navigation

## Overview
Replace the manual element-by-element Markdown assembly with Docling's native per-page Markdown export (`doc.export_to_markdown(page_no=N)`), store per-page Markdown data on disk, add new API endpoints to serve individual page Markdown, build a page-by-page Markdown viewer in the frontend (mirroring the PDF pane's page navigation), and fix bottom padding/overflow issues on the document viewer page.

## Feature Requirements

### Must-Have
- [ ] Backend generates per-page Markdown using Docling's native `doc.export_to_markdown(page_no=N)` with `ImageRefMode.EMBEDDED` for base64 inline images
- [ ] Per-page Markdown stored as individual files (`page_1.md`, `page_2.md`, ...) in a `markdown_pages/` subdirectory under the job directory
- [ ] Full-document Markdown file (`result.md`) still generated by concatenating all page Markdown (backward compatibility for download endpoint)
- [ ] New API endpoint `GET /api/jobs/{job_id}/markdown/pages` returns page count metadata
- [ ] New API endpoint `GET /api/jobs/{job_id}/markdown/pages/{page_number}` returns a single page's Markdown content
- [ ] MarkdownTab shows page-by-page navigation with prev/next buttons and page indicator (matching PdfPane's navigation style)
- [ ] Each page's Markdown renders independently with ReactMarkdown + remark-gfm
- [ ] Fix `ResultsViewer` hardcoded `max-h-[600px]` to use flex-based height that fills available space
- [ ] Fix `JobDetailPage` bottom padding so split pane content is not clipped

### Nice-to-Have
- [ ] Keyboard shortcuts (left/right arrow) for Markdown page navigation when tab is focused
- [ ] Page number input field for direct jump
- [ ] Prefetch adjacent pages (N-1, N+1) after current page loads for snappy navigation
- [ ] Fallback: if Docling's native export fails for a page, fall back to current manual assembly for that page

## Technical Design

### Data Models

**ParsedResult model** -- add one new column:

| Column | Type | Description |
|--------|------|-------------|
| `markdown_pages_dir` | `String(512), nullable` | Path to directory containing per-page `.md` files |

No new tables needed. Per-page Markdown files live on disk to avoid bloating the SQLite database with large text blobs (base64 images can be several MB per page).

### File Storage Layout

```
uploads/{job_id}/
  input.pdf
  images/
  result.json
  result.md              # full document (concatenated from pages)
  result.txt
  chunks.json
  markdown_pages/        # NEW
    page_1.md
    page_2.md
    ...
```

### API Endpoints

| Method | Path | Auth | Request Body | Response |
|--------|------|------|-------------|---------|
| GET | `/api/jobs/{job_id}/markdown/pages` | No | - | `{ "job_id": str, "total_pages": int, "pages": [1, 2, ...] }` |
| GET | `/api/jobs/{job_id}/markdown/pages/{page_number}` | No | - | `{ "job_id": str, "page_number": int, "total_pages": int, "content": str }` |

Existing endpoints remain unchanged:
- `GET /api/jobs/{job_id}/download/markdown` still serves the full `result.md` file

### Backend Changes

**`src/backend/services/parser.py`**:
- After `converter.convert()`, iterate pages 1..total_pages calling `doc.export_to_markdown(page_no=N)` (with `image_mode=ImageRefMode.EMBEDDED` if supported, otherwise default)
- Store the Markdown strings in a new key on the result dict: `"page_markdowns": { "1": "...", "2": "...", ... }`
- This key is used by the exporter to write per-page files; it is stripped before saving `result_json` to avoid DB bloat

**`src/backend/services/exporter.py`**:
- New function `to_page_markdowns(result: dict, pages_dir: Path) -> Path` writes each entry from `result["page_markdowns"]` to `pages_dir/page_{N}.md`
- Modify `to_markdown()` to concatenate per-page Markdown with `---\n## Page {N}\n` separators (keeping the existing format users see in full download)
- Update `generate_all_exports()` to call `to_page_markdowns()` and return `markdown_pages_dir` in the paths dict

**`src/backend/routes/results.py`**:
- Add `GET /api/jobs/{job_id}/markdown/pages` handler: reads `markdown_pages_dir` from ParsedResult, lists `.md` files, returns total_pages and page number list
- Add `GET /api/jobs/{job_id}/markdown/pages/{page_number}` handler: reads `markdown_pages/page_{N}.md` from disk, returns content as JSON. Returns 404 if page file does not exist.

**`src/backend/routes/upload.py`**:
- In `_run_parse_job()`, pass `markdown_pages_dir` from `export_paths` into the `ParsedResult` constructor
- Strip `page_markdowns` from the result dict before serializing to `result_json` (to avoid storing duplicate large data in DB)

**`src/backend/models/result.py`**:
- Add `markdown_pages_dir: Mapped[str | None] = mapped_column(String(512), nullable=True)`

### Frontend Changes

**`src/frontend/src/services/api.ts`**:
- Add `MarkdownPagesResponse` interface: `{ job_id: string, total_pages: number, pages: number[] }`
- Add `MarkdownPageResponse` interface: `{ job_id: string, page_number: number, total_pages: number, content: string }`
- Add `getMarkdownPages(jobId: string): Promise<MarkdownPagesResponse>`
- Add `getMarkdownPage(jobId: string, pageNumber: number): Promise<MarkdownPageResponse>`

**`src/frontend/src/components/tabs/MarkdownTab.tsx`** (rewrite):
- State: `currentPage`, `totalPages`, `pageContent`, `loading`, `error`
- On mount: call `getMarkdownPages(jobId)` to get `totalPages`
- On page change: call `getMarkdownPage(jobId, currentPage)` to fetch content
- Render: navigation header bar (matching PdfPane style) + scrollable ReactMarkdown area
- Keep all existing ReactMarkdown component overrides (headings, tables, code, images, blockquotes, etc.)
- Fallback: if the pages API returns 404 (old job without per-page Markdown), fall back to fetching full document via existing download endpoint and display without pagination

**`src/frontend/src/components/ResultsViewer.tsx`**:
- Line 332: replace `max-h-[600px]` with `flex-1 min-h-0` so the content area fills available vertical space within the flex column

**`src/frontend/src/pages/JobDetailPage.tsx`**:
- Line 138: the split pane container `<div className="flex-1 min-h-0 px-4 pb-4">` -- the `pb-4` (16px) causes clipping when combined with the header. Change to `pb-2` or remove, and ensure the parent `flex flex-col h-screen overflow-hidden` correctly distributes space.

## Design Direction

- **Mood / Identity**: Developer-tool utility. The Markdown viewer should feel like a clean document reader -- Notion's reading view or Typora's preview mode. Functional, content-forward, not decorative.
- **Color direction**: Cool neutral grays with blue accents, consistent with the existing codebase palette. The Markdown navigation bar matches the PDF pane's `bg-white border-b` header style for visual symmetry across the split pane.
- **Reference style**: "VS Code panel" -- compact chrome, content-forward, minimal ornamentation. Navigation controls are small and unobtrusive.
- **Anti-patterns**: No floating page selectors or modal overlays. No page transition animations. No separate page list sidebar in the Markdown tab -- just prev/next controls like the PDF pane. No skeleton loaders -- use simple text indicators matching existing loading patterns.
- **Key UX moments**:
  - Page transition in Markdown tab should feel instant -- the navigation bar responds immediately (updates page number), content loads shortly after
  - The Markdown and PDF navigation bars should be visually aligned when both panes are visible side-by-side, reinforcing the "two views of the same document" mental model
  - Empty/error states use the same patterns as existing tabs (small gray or red text, no large illustrations)

## Tech Stack Decisions

No deviations from the existing stack. All changes use libraries already in the project:
- **Docling** `doc.export_to_markdown(page_no=N)` is available in the installed version
- **ReactMarkdown** + **remark-gfm** already used in the current MarkdownTab
- **Tailwind CSS** for all styling

**Decision: Disk storage over DB storage for per-page Markdown**. Markdown with embedded base64 images can be several MB per page. Storing on disk avoids SQLite row size issues and keeps the DB lean. Files are colocated with other exports in the job directory.

**Decision: No Alembic migration**. The project uses `Base.metadata.create_all()` for schema management (no Alembic found). The new nullable column is added via the same pattern. Existing rows get `NULL` for `markdown_pages_dir`, handled gracefully by returning 404 from the page API.

**Decision: Docling `export_to_markdown` parameters**. If `page_no` parameter is not supported in the installed Docling version, fall back to calling `doc.export_to_markdown()` for the full document and splitting the output by page boundaries. The spec prefers the per-page API but the implementation should handle both cases.

## Risks

- **Docling `export_to_markdown(page_no=N)` signature may differ**: Mitigation -- check Docling source at runtime. If `page_no` kwarg is not available, export full document Markdown and split by page, or fall back to the existing manual assembly per page.
- **Base64 images inflate per-page Markdown size**: Mitigation -- acceptable for single-page API responses. Full-document download was already large. Could add `?embed_images=false` query param in a future iteration.
- **Existing jobs lack per-page Markdown**: Mitigation -- API returns 404, frontend falls back to full-document fetch. Users can reprocess to generate per-page files.
- **SQLite column addition on running system**: Mitigation -- nullable column with default NULL is safe for SQLite `CREATE TABLE IF NOT EXISTS` semantics. No data migration needed.
- **GFM table quality from Docling native export vs manual assembly**: Mitigation -- Docling's native export is expected to produce equal or better table formatting. If quality regresses, the manual assembly remains available as a fallback in `exporter.py`.
